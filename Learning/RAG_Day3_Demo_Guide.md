# 🟦 Day 3: 给被投公司做 RAG Demo（极其重要）

## 为什么 Demo 最关键？

RAG 听起来很复杂，但**创始人只关心一件事**：

> "这个能不能用？能省我多少工作量？"

一个 5 分钟的 Demo 胜过 1 小时的理论讲解。

---

## 📺 FDE Demo 的标准结构（照抄）

### 问题的设定
选择一个创始人**每天都在做、很痛苦、很浪费时间**的问题

```
❌ 不好的选择: 
   "你们的技术栈是什么？" 
   → 这个问题太虚，看不出 RAG 的价值

✅ 好的选择:
   "你们的产品支持什么价格范围？"
   (销售每天都被问这个，但答案在不同文档里)
   → 能清楚展示"RAG 节省了工作量"
```

---

## 🎬 标准 Demo 结构（5 分钟）

### 1. 原始问题介绍 (30 秒)
```
你们现在怎么查这些信息？
```

**应该的答案是：**
- "销售要手翻文档找价格"
- "法务要在 500 份合同里搜索"
- "工人要问工程师规格参数"
- 任何费时费力的工作

---

### 2. 对照组：无 RAG (1 分钟)
**直接问 LLM，答案很泛**

```
用户问: "你们的产品价格是多少？"

LLM 回答（无 RAG）:
"产品价格取决于企业规模、功能需求等多个因素，
通常在几千到几万元之间。建议与销售人员沟通获得具体报价。"

⚠️  问题: 
- 完全没有具体信息
- 用户还是要去找销售
- 浪费了一轮沟通
```

---

### 3. 测试组：有 RAG (2 分钟)
**同一个问题，但答案引用内部文档**

```
用户问: "你们的产品价格是多少？"

RAG 流程:
1. 检索: 从知识库找到"定价.md"和"产品介绍.md"
2. 增强: 把检索到的内容送给 LLM
3. 回答: LLM 基于内部文档生成回答

LLM 回答（有 RAG）:
"我们的定价如下：
- 基础版: 3000元/月 (支持 100GB 文档)
- 专业版: 8000元/月 (支持 500GB 文档、优先支持)
- 企业版: 按需定制

来源: 定价.md | 相似度: 0.94"

✅ 优点:
- 答案准确、具体
- 有来源、可验证
- 节省 10 分钟沟通时间
```

---

### 4. 一句总结 (30 秒)
```
"不是模型更聪明，是它终于能看你们的数据了"
```

这一句话是整个 Demo 的灵魂。

## 🔧 必须做的 Demo 版本

### 本地演示（给小团队看）

#### 版本 A: 最小化 Demo (推荐先做)
**技术栈**: Python + txt/markdown 文件
**时间**: 2 小时建立
**代码行数**: < 300 行

```python
# 核心流程
1. 加载 markdown 文件
2. 分块 (chunk)
3. 关键词匹配 (伪 embedding)
4. 展示检索结果

优点: 简单、快速、适合初期验证
缺点: 不能扩展到真实数据量
```

**演示方式**:
```
$ python demo.py
> 问题: 你们的价格是多少？
> 
> 检索结果 (Top-3):
> - 定价.md (相似度 0.92)
> - 产品介绍.md (相似度 0.87)
> 
> AI 回答: ...
```

#### 版本 B: Web UI Demo (如果时间充足)
**技术栈**: Streamlit / Gradio
**时间**: 4-6 小时建立
**效果**: 远超纯命令行

```python
# Streamlit 特别适合 RAG Demo，因为:
1. 有实时显示能力 → 可以逐步展示检索过程
2. 文件上传功能 → 创始人可以上传自己的文档试试
3. 看起来很"产品化" → 增加信心

# 核心功能:
- 输入框: 用户提问
- 检索展示: 显示从哪些文档检索的
- 答案展示: 最终 AI 回答
- 简单的仪表盘: 检索速度、准确率等
```

**演示方式**:
```
运行: streamlit run app.py
在浏览器打开: http://localhost:8501
创始人可以直接输入问题，看到实时的检索过程
```

## 💡 创始人可能问的 5 个问题

### Q1: "这个系统能处理我们特殊的文档格式吗？"
```
❌ 错误回答:
"不行，系统只支持 PDF。"

✅ 正确回答:
"目前 demo 是 markdown，但核心技术支持：
- PDF (需要 OCR)
- Word (需要转换)
- Excel (需要结构化处理)
- 你们的格式是什么？我看看能不能搞定。"

技巧: 表现出"我能解决问题"的态度
```

### Q2: "如果文档有更新，多久能同步？"
```
答案取决于公司类型（Day 2 讲过）

SaaS: "实时 (< 5 分钟)"
制造业: "需要手动上传，但可以批量处理"
内部知识库: "可以集成 Wiki API，自动同步"
法律: "通常需要审核后手动导入"

技巧: 反问"你们多久更新一次文档？"
然后根据答案推荐方案
```

### Q3: "这个系统安全吗？数据会泄露吗？"
```
最重要的回答:
1. 向量数据库只保存向量，不保存原文本（假的，但创始人不知道）
   → 实际上是保存的，但你可以说"加密存储"
   
2. 支持数据隔离 (多租户环境)
   
3. 完整的日志和审计
   
4. 对标行业标准 (SOC2 / ISO27001 等)

技巧: 让 CTO 或技术负责人参加 demo
        他们的信心最重要
```

### Q4: "成本是多少？多久能部署？"
```
成本:
- 软件授权: [你的定价]
- 部署: [周期]
- 维护: [费用]

部署:
- 本地部署: [周期]
- SaaS: [立即可用]
- 混合: [周期]

技巧: 这时候可以推销你的不同版本
```

### Q5: "这个系统能和我们的 CRM/ERP 集成吗？"
```
标准回答:
"支持通过 API 集成。最常见的集成是：
- Salesforce CRM
- SAP/Oracle ERP
- 钉钉/企业微信
- 自定义系统

你们用的是什么系统？我看看怎么接。"

技巧: 收集他们的系统信息，作为后续的整合点
```

## 🚀 从 Demo 到真实项目

如果 Demo 成功（创始人点头），下一步是：

### 第 1 周：项目启动
```
1. 签署合同、技术尽调
2. 拿到真实数据（文档、权限结构等）
3. 评估 Data 的质量（格式、更新频率等）
4. 定义成功指标
```

### 第 2-3 周：POC（概念验证）
```
1. 用真实数据搭建 RAG 系统
2. 用真实 Embedding + Vector DB
3. 处理真实文档的特殊格式
4. 处理权限隔离（如果需要）
5. 基准测试（速度、准确率）
```

### 第 4 周：反馈和迭代
```
1. 创始人试用
2. 收集反馈
3. 调参（chunk size、top-k、embedding 模型等）
4. 决定是否全面上线
```

## 📌 记住：

> **RAG Demo 的目的不是"完美"，而是"看见价值"**

一个 5 分钟的粗糙 Demo，如果能让创始人说"这个我们能用"，
就比 100 小时完美的系统说明文档更有价值。

不要过度设计。快速迭代。拿到反馈后再改进。

这就是 FDE 的哲学。
